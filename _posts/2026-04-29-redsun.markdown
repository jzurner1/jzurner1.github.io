RedSun is a recently released Windows Defender exploit that provides local privilege escalation. [Initially released by a researcher peeved with Microsoft's Security Response Center](https://github.com/Nightmare-Eclipse/RedSun), it has since been seen [exploited in the wild](https://www.huntress.com/blog/nightmare-eclipse-intrusion). While there are plenty of explanations on how it works, I haven't seen any actual code breakdowns, and this seems like a good learning opportunity. As a reminder, the core of the exploit is that Defender trusts whatever file path is used during a rollback, not checking if a junction is in place.

One more note - there are no new revelations in this blog post; I am just breaking down the content of the exploit that has been available for a couple weeks now. I might follow up with a deeper dive into the relevant mechanics of Defender, but that depends on if I have time and if I find something else more interesting. Also, if you see any mistakes here, please reach out! I'm always looking to improve on my understanding and I'd be very appreciative.

## Exploit Process
- The exploit creates a named pipe (`\\??\pipe\REDSUN`) that will be used towards the end to relay the user's session ID to the elevated process
- A staging directory is created at `C:\Users\<username>\AppData\Local\temp\RS-<guid>\`
- A background thread (`ShadowCopyFinderThread`) is launched, which snapshots all existing `\Device\HarddiskVolumeShadowCopy*` entries as a baseline and starts polling for new ones
- The EICAR test string is written to `C:\Users\<username>\AppData\Local\temp\RS-<guid>\TieringEngineService.exe`
- That file is opened with `FILE_EXECUTE`, which triggers a scan by Windows Defender; the EICAR string is detected and Defender begins remediation
- The first step of Defender's remediation process is to create a volume shadow copy of the full volume that contains the malicious file
- The background thread (`ShadowCopyFinderThread`) detects the new VSS entry
- It then opens the version of `TieringEngineService.exe` that is stored within the shadow copy and places a batch OPLOCK on it. Note that the OPLOCK is preemptive and will be used to notify when Defender accesses the file
- Finally, the background thread signals that the shadow copy was created, found, and an OPLOCK has been placed on it
- The main function receives the signal and deletes the EICAR file
- `DoCloudStuff()` is called, which registers the staging directory as a cloud files sync root and creates a cloud placeholder called `TieringEngineService.exe`, making Defender think the file belongs to a cloud sync provider
- Because the file has a cloud files attribute, Defender enters the rollback code path instead of a simple delete, trying to restore the file to its pre-infection state using the VSS snapshot, similar to rolling back a document on Google Docs.
- Main signals the background thread that pre-work (file deletion and cloud setup) is complete, then waits for the thread to finish
- The background thread's shadow copy OPLOCK eventually breaks (Defender accesses the shadow copy), the thread sees main's signal, releases its shadow copy handle, and wakes main
- Main sets aside the staging directory (appends `.TMP`) and creates a new version with a new `TieringEngineService.exe` set to read-only
- A second OPLOCK is created on the new file, followed by a file mapping with a section object, which prevents Defender from deleting or overwriting the file directly
- Main waits until this second OPLOCK breaks, indicating Defender's remediation opened the new file
- Main releases the file mapping and removes the file from the directory
- A directory junction is created in the now-empty staging directory, pointing to `C:\Windows\System32`
- Defender writes the restored file to the original detection path, which now resolves through the junction to `C:\Windows\System32\TieringEngineService.exe`
- Main runs a loop attempting to supersede the file; when it succeeds, the attacker becomes the file's creator
- Main closes the handle, then writes a copy of the exploit binary to `C:\Windows\System32\TieringEngineService.exe`
- Main calls `LaunchTierManagementEng`, which instantiates the Storage Tiers Management Engine COM object, which runs `TieringEngineService.exe` as SYSTEM
- The now-running exploit detects that it is running as SYSTEM and uses the named pipe to create a SYSTEM shell for the user

An interesting side effect of mapping a file into memory is that a section object is created, backed by the file. When a section object exists on a file, it cannot be deleted, overwritten, or opened (in a way that would conflict with the mapping), even for SYSTEM processes like Windows Defender.

## LaunchConsoleInSessionId()
The goal of this function is to run an interactive console with SYSTEM privileges. Normally code run as SYSTEM is in session 0, which is an isolated and non-interactive session; obviously that doesn't make for a very good interactive session. The solution is to create a SYSTEM token (technically a duplicate) and stick the user's current session ID onto it, making it accessible.

```c
    HANDLE hpipe = CreateFile(L"\\??\\pipe\\REDSUN", GENERIC_READ, NULL, NULL, OPEN_EXISTING, FILE_ATTRIBUTE_NORMAL, NULL);
    if (hpipe == INVALID_HANDLE_VALUE)
        return;
	
	DWORD sessionid = 0;
    if (!GetNamedPipeServerSessionId(hpipe, &sessionid))
        return;
    CloseHandle(hpipe);
```
First, a handle is created to an existing file path, specifically to the named pipe created earlier. The session ID of the pipe server is retrieved (representing the user that ran the exploit), then the connection is closed.

```c
    HANDLE htoken = NULL;
    if (!OpenProcessToken(GetCurrentProcess(), TOKEN_ALL_ACCESS, &htoken))
        return;
	
	HANDLE hnewtoken = NULL;
    bool res = DuplicateTokenEx(htoken, TOKEN_ALL_ACCESS, NULL, SecurityDelegation, TokenPrimary, &hnewtoken);
    CloseHandle(htoken);
    if (!res)
        return;
```
Next a handle is opened to the current process with all permissions, which is then duplicated to `hnewtoken`. The original is closed. Note that a token actively being used by a process can't be modified, hence why a duplicate was made.

```c
    res = SetTokenInformation(hnewtoken, TokenSessionId, &sessionid, sizeof(DWORD));
    if (!res)
    {
        CloseHandle(hnewtoken);
        return;
    }
```
The newly duplicated token is then set to have the session ID of the originally opened named pipe, ensuring it will be accessible by the user.

```c
    STARTUPINFO si = { 0 };
    PROCESS_INFORMATION pi = { 0 };
    CreateProcessAsUser(hnewtoken, L"C:\\Windows\\System32\\conhost.exe", NULL, NULL, NULL, FALSE, NULL, NULL, NULL, &si, &pi);

        CloseHandle(pi.hProcess);
    if (pi.hThread)

```
Finally, an instance of `conhost.exe` is created under the session ID of the user, resulting in a SYSTEM-level console accessible to the regular user.
